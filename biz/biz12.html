IT Change Management Discussion

[Discussion material in English. Based on GTAG “IT Change Management (3rd Edition)”.]

Core idea: Change management is a repeatable and auditable way to protect the production environment while implementing technology changes (including patches). It reduces unplanned work, limits unintended outcomes, and improves service quality.

What “Change Management” Covers (GTAG Summary)

In IT, change management is the systematic process to manage enhancements, updates, installations, fixes, configuration changes, and patches to production systems — while minimizing disruption and controlling risk.

Repeatable process Approval & authorization Segregation of duties Migration across environments Patches as standard changes Metrics & continuous improvement

Configuration Management — Discussion Points

Even though the GTAG explicitly says it does not deep-dive into configuration management, effective change management still depends on configuration discipline: knowing what exists, what changed, who changed it, and when.

Why Config Mgmt Matters

  • Traceability: Link each production change to an approved request/ticket.
  • Version control: Ensure only approved versions of code/config are promoted.
  • Baseline & rollback: Maintain known-good baselines to restore service quickly.
  • Auditability: Evidence that changes are authorized and not manipulated.
Config Mgmt Control What It Prevents Audit / Risk Angle
Version control (repo) Untracked code/config edits Supports “who/what/when” and change approvals
Baselines & standard builds Drift across servers/environments Reduces unexpected outcomes when migrating changes
Immutable logs / evidence Tampering after deployment Critical for proving integrity during audits
Rollback/backout plan Extended outages from failed changes GTAG stresses rollback planning as part of the process

What Does (DEV → TEST → UAT → PROD) Mean?

DEV → TEST → UAT → PROD is the standard environment flow that IT changes (code, configuration, patches) follow before reaching the live system. It’s the migration path used to protect the production environment.

DEV (Development)
Where changes are created
  • Build and modify code/config
  • Frequent changes & experimentation
TEST (Testing)
Where changes are validated technically
  • Unit / integration / regression tests
  • Defects found before business testing
UAT (User Acceptance Testing)
Where business confirms requirements
  • Business users validate outcomes
  • Example: finance checks postings/reports
PROD (Production)
The real, live environment
  • Used by real users/customers
  • Errors impact operations & reporting
Why is this flow important?
  • Separating environments reduces outage, integrity, and security risk.
  • Skipping gates (DEV → PROD) increases the chance of untested/unauthorized changes.
  • Each step acts as a control gate to ensure:
    • the change is properly tested
    • the change is formally approved
    • the change is deployed by authorized personnel only
  • This supports segregation of duties (SoD) and auditability.

Migration (DEV → TEST → UAT → PROD) — Discussion Points

GTAG highlights that systems typically have multiple environments (DEV/TEST/UAT/PROD), and the controlled movement of a change across environments is called migration. A key control is segregation of duties so developers cannot directly deploy to production.

Good Migration Practices

  • Environment consistency: Keep TEST/UAT as identical as possible to PROD (versions, patches, configs).
  • SoD in promotion: Developers build changes; an independent role (e.g., release manager) migrates to PROD.
  • Pre-production testing: Mandatory evidence of testing before PROD approval.
  • Verification after deployment: Confirm success + confirm the process was followed.
Step Migration Gate Typical Evidence
DEV → TEST Basic technical validation Build logs, unit tests, peer review
TEST → UAT Business readiness Test results, defect closure, UAT plan
UAT → PROD Formal authorization & SoD CAB approval, deployment ticket, access profiles
Post-PROD Validate impact Monitoring, PIR (post-implementation review), outage analysis

Patches as Part of Change Management

GTAG treats patches as a category of change that should follow the normal change management process. Patches can be urgent (security vulnerabilities), complex, and sometimes “pushed” automatically by vendors — which adds risk.

Patch reality check: Even “successful” patch deployment can cause unintended outages. That’s why GTAG recommends testing patches in a staging/sandbox environment when possible, and managing them in a planned cycle.
  • Risk-based prioritization: Critical vulnerabilities need faster timelines.
  • Bundle & schedule: More efficient than patching one-by-one without coordination.
  • Vendor push awareness: IT should know when patches may auto-apply and plan monitoring.

Practical Dialogue: Migration & SoD Failure (DEV Direct to PROD)

IT Auditor Planning
Can you walk me through how a change moves from DEV to PROD? Who is allowed to migrate changes into production?

Developer Reality
I develop in DEV and sometimes push the release directly to PROD when we’re under time pressure. It’s faster.

IT Auditor Control Risk
That breaks segregation of duties. If the same person develops and deploys, an unauthorized or untested change can reach production without detection. Do you have monitoring controls to reconcile actual PROD changes to approvals?

IT Ops / Release Manager Gap
We don’t have a formal release manager role. Also, we don’t consistently reconcile PROD deployments against CAB approvals.

IT Auditor Finding
Then the organization cannot confidently claim the change process is controlled or auditable. Remediation usually includes:

  • Remove developer write/deploy access to PROD (least privilege)
  • Independent migration role (release/change manager) or automated pipeline with approvals
  • Mandatory testing evidence before PROD promotion
  • Detective control: monitor PROD and reconcile changes to approved tickets
  • Required rollback/backout plans and post-implementation review for significant changes

Link to “식통감기”: DEV direct-to-PROD usually fails all four: weak Test, weak Control, weak Audit, and weak Record.

Metrics to Judge Process Health

GTAG recommends using objective indicators (dashboards) to judge whether change management works. Common examples:

  • Change success rate (changes not causing outages / service impairment)
  • Unauthorized change count (target should be zero)
  • Emergency change ratio (including emergency patches)
  • Patch latency: average time from vendor release to deployment
  • Unplanned work %: time spent on break/fix caused by failed changes

Discussion Questions

  • In your organization, what is the biggest weakness: Test, Control, Audit, or Record?
  • Do developers have any way to deploy or modify configurations directly in PROD? If yes, what compensating controls exist?
  • How do you ensure environments are “as identical as possible” (versions/patches/configs) before migration to PROD?
  • How are patches handled: scheduled cycle vs ad hoc emergency? How do you handle vendor auto-push patches?
  • What evidence would you show an auditor to prove that a change was properly approved, tested, and migrated?
  • What metrics do you track today, and do they actually drive improvement (or just reporting)?
