ITAC Risks at the Input Stage

AI OCR in Accounting Automation

[Study material for accounting professionals. Technical details are simplified for discussion purposes.]

Many companies now use AI-based OCR to read invoices and automatically create journal entries in their accounting systems. While this improves efficiency, it also introduces new IT Application Control (ITAC) risks, especially at the input stage.

Why the Input Stage Matters

The input stage includes:

  • Invoice upload
  • OCR data extraction
  • Document type classification

If data is incorrect at this stage, downstream reviews may not detect the error. This is why auditors often say:

“Garbage in, garbage out.”

Key Input-Stage Risks (ITAC Focus)

Risk Area What Can Go Wrong Accounting Impact
OCR Recognition Errors Amount, date, vendor, tax, or currency is misread by OCR Incorrect journal entries
Misstated expenses or liabilities
Lack of Input Validation OCR data flows directly into ERP without logic checks Negative amounts or invalid tax postings

EBITDA distortion

✔ Auditors expect controls such as OCR confidence thresholds, mandatory human review, and document integrity checks.

Why Auditors Treat This as ITAC

These risks are considered ITAC because:

  • Controls are embedded in the system
  • Errors occur before human review
  • Financial accuracy depends on system logic

Practical Dialogue: Finance Lead vs. OCR Developer

Finance Lead Accounting / Audit
We are using OCR to automatically read invoices and post entries into the ERP. From an audit perspective, my main concern is simple: how do we know the data going into the system is reliable? If the input is wrong, everything after that can also be wrong.

OCR Developer System Flow
That makes sense. The OCR process has several steps. First, the invoice is uploaded. Then the system reads the text and tries to identify key fields like vendor name, invoice date, amount, and tax. For each field, the system also gives a confidence score, meaning how sure it is that the value is correct.

Finance Lead Risk Awareness
Confidence scores sound helpful, but from experience, I know that numbers can still look “reasonable” and be wrong. For example, the amount might be correct, but the tax could be missing or calculated incorrectly. Do we have any checks beyond just OCR confidence?

OCR Developer Basic Validation
Yes, we do. Before sending data to the ERP, the system runs basic validation rules. For example:

  • The total amount cannot be negative unless it is a credit note
  • Tax amount must roughly match the net amount and tax rate
  • The posting date must fall within an open accounting period
  • The vendor must exist in the vendor master
If any of these checks fail, the invoice is stopped for manual review.

Finance Lead Audit Trail
That’s good. Another thing auditors always ask for is evidence. They want to see the original invoice, what the OCR system extracted, and whether anyone changed the data manually. Can we track all of that?

OCR Developer Traceability
Yes. The original invoice file is stored and cannot be overwritten. If someone edits an OCR result, the system records who changed it, when, and what was changed. The final data sent to the ERP is also saved, so we can always trace a journal entry back to the source document.

Finance Lead Controls & Roles
What about responsibilities? From a controls point of view, the person uploading an invoice should not be the same person approving it. Otherwise, we may have segregation of duties issues.

OCR Developer User Roles
We thought about that as well. The system separates roles: one user uploads invoices, another reviews OCR results, and a different user approves posting to the ERP. The system does not allow one person to do all steps alone.

Finance Lead Classification Risk
I’m also concerned about document type. If an invoice is mistakenly treated as a credit memo, or if capital expenditure is treated as an expense, that directly affects financial reporting. How do we reduce that risk?

OCR Developer Practical Safeguards
For high-risk cases, the system is more conservative. If the document type is unclear, or the amount is unusually high, the invoice is automatically sent for manual review. We also regularly review errors and adjust the rules so the system improves over time.

Finance Lead Conclusion
That helps a lot. From an audit perspective, as long as we can show input checks, clear approval steps, and a full audit trail, auditors are more likely to accept OCR as a controlled process rather than a black box.

✔ This type of conversation reflects how companies typically explain OCR-related controls to auditors: focusing less on complex technology and more on practical risks, checks, and responsibilities.
