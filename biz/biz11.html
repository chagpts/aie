ITAC Risks at the Input Stage

AI OCR in Accounting Automation

[Study material for accounting professionals. Technical details are simplified for discussion purposes.]

Many companies now use AI-based OCR to read invoices and automatically create journal entries in their accounting systems. While this improves efficiency, it also introduces new IT Application Control (ITAC) risks, especially at the input stage.

Why the Input Stage Matters

The input stage includes:

  • Invoice upload
  • OCR data extraction
  • Document type classification

If data is incorrect at this stage, downstream reviews may not detect the error. This is why auditors often say:

“Garbage in, garbage out.”

Key Input-Stage Risks (ITAC Focus)

Risk Area What Can Go Wrong Accounting Impact
OCR Recognition Errors Amount, date, vendor, tax, or currency is misread by OCR Incorrect journal entries
Misstated expenses or liabilities
Lack of Input Validation OCR data flows directly into ERP without logic checks Negative amounts or invalid tax postings
Source Document Integrity Uploaded documents can be modified or replaced without trace Unsupported or fraudulent entries
Document Misclassification Invoice classified as credit memo
CapEx treated as OpEx 		Wrong account posting
EBITDA distortion

✔ Auditors expect controls such as OCR confidence thresholds, mandatory human review, and document integrity checks.

Why Auditors Treat This as ITAC

These risks are considered ITAC because:

  • Controls are embedded in the system
  • Errors occur before human review
  • Financial accuracy depends on system logic

Practical Dialogue: Finance Lead vs. AI OCR Engineer

Finance Lead (재무 팀장) ITAC / 감사 관점
지금 OCR로 인보이스를 자동전표로 넘기는 구조잖아요. 감사인이 제일 먼저 물어보는 게 “OCR 추출값이 ERP에 들어가기 전 어떤 애플리케이션 컨트롤이 있냐”예요. 특히 완전성(Completeness), 정확성(Accuracy), 유효성(Validity)을 어떻게 보장하죠?

AI OCR Engineer (OCR 개발자) Pipeline / Model
현재 파이프라인은 Document Ingestion → Pre-processing(De-skew/De-noise) → Layout Analysis (텍스트 블록/테이블/키-값 영역 분리) → Text Recognition → Key-Value Extraction → Document Classification → ERP Posting API 순서예요. 각 단계마다 confidence score를 뽑고, 필드 단위로 threshold 아래면 human-in-the-loop 큐로 라우팅합니다.

Finance Lead 통제 설계
confidence threshold만으로는 부족하다고 보더라고요. 예를 들어 금액(Amount)이 숫자로는 맞는데 소수점이 밀리거나, 세액(VAT)과 공급가(넷) 매핑이 뒤집히는 케이스요. 비즈니스 룰 기반 validation은 어디서 돌고 있나요?

AI OCR Engineer Validation / Rule Engine
맞아요. 그래서 OCR score는 “인식 품질”이고, 회계적으로 맞는지는 별도 검증이 필요해요. 우리는 Rule Engine에서 필드 논리검증을 추가했습니다. 예를 들면:

  • Amount sanity check: 음수 금액 허용 여부(크레딧 메모만 허용), 상한/하한 범위
  • VAT cross-check: 공급가 × 세율 ≈ 세액(허용 오차/반올림 로직 포함)
  • Currency validation: 통화코드(ISO) + 환율 테이블 매칭, 환산금액 재계산
  • Date validation: 전기기간(Posting period) 밖이면 에러/승인필수
  • Vendor master match: 벤더명 fuzzy match + 사업자번호(또는 Tax ID) exact match 우선
룰 통과 못 하면 ERP로 POST 못 하고 예외 처리로 빠져요.

Finance Lead 감사 Trail
좋네요. 그럼 감사인이 요구하는 “증적”은요? 입력부터 전표 생성까지 audit trail이 있어야 해요. 원본 인보이스, OCR 결과 JSON, 수정 이력(누가/언제/무엇을), 그리고 ERP에 실제로 POST된 payload까지요. 그리고 원본 문서가 중간에 바뀌면 큰 문제고요.

AI OCR Engineer Immutability / Logging
그 부분은 문서 무결성(Document Integrity) 통제로 잡았습니다. 업로드 시점에 원본 파일에 대해 SHA-256 해시를 생성하고, OCR 결과/전표 payload/사용자 수정 로그에 동일한 Document ID로 연결해요. 저장은 WORM(Write Once Read Many) 옵션 또는 Object Storage 버전닝으로 “원본 불변성”을 유지합니다. 또 append-only event log 형태로 Field-level change history를 남기고, 사용자 수정은 전자서명(Approval) 워크플로우를 통과해야 ERP로 나가게 했어요.

Finance Lead 권한 / SoD
권한도 중요해요. 업로드한 사람이 검증도 하고 승인도 하면 SoD(Segregation of Duties) 위반으로 보일 수 있어요. 그리고 개발자가 운영환경에서 룰을 바꾸거나 모델을 교체하면 Change Management 이슈죠. 그건 어떻게 관리해요?

AI OCR Engineer IAM / Change Control
권한은 RBAC(Role-Based Access Control)로 분리했고, Uploader / Reviewer / Approver 역할을 분리해 동일인이 연속 단계 수행 못 하도록 정책을 걸었어요. 시스템 쪽은

  • Model versioning: 모델/프롬프트(있다면)/룰셋을 버전으로 고정
  • CI/CD 승인 게이트: PR 승인 + 배포 승인(승인권자 분리)
  • Config as Code: 룰 변경도 Git으로 추적하고 릴리즈 노트 생성
  • Prod 접근 통제: 개발자는 운영 DB 직접 수정 불가, break-glass는 로그/승인 필수
감사 대응용으로 변경 전/후 비교(diff), 배포 이력, 승인자까지 자동으로 남겨요.

Finance Lead 분류 리스크
문서 분류(Document Classification)도 걱정이에요. 인보이스를 크레딧 메모로 분류하면 전표 부호가 뒤집히고, CapEx/OpEx 분류가 틀리면 계정과목이 바뀌죠. 모델이 “그럴듯하게” 틀리면 더 위험해요. 오분류 통제는 어떻게 하죠?

AI OCR Engineer Classifier / Quality Monitoring
분류는 단일 모델에만 의존하지 않고 앙상블(Ensemble) + 룰 기반 가드레일을 섞었어요. 예를 들어 “Credit Memo”는 negative total, “credit note”, “refund” 같은 keyword + 금액 패턴도 같이 봅니다. 또 분류 confidence가 낮거나 특정 고위험 벤더(고액/빈번)는 mandatory review로 보내요. 운영에서는 drift monitoring (벤더 템플릿 변경 등)으로 성능 저하를 탐지하고, 샘플링해 precision/recall을 주기적으로 리포팅합니다.

Finance Lead 감사 결론
좋아요. 결국 감사에서는 1) 입력값 통제(Validation), 2) 원본 무결성, 3) 권한 및 SoD, 4) 변경관리, 5) 예외 처리와 샘플링 모니터링 이 다섯 가지를 “시스템 통제”로 보여주면 ITAC으로 설득이 되겠네요.

✔ 위 대화에서 나온 키워드(예: confidence threshold, rule engine, SHA-256, WORM, RBAC, CI/CD 승인 게이트, model versioning, drift monitoring)는 감사인의 ITAC 테스트 시 통제 설계(Design)와 운영 효과성(Operating Effectiveness) 설명에 자주 사용됩니다.
