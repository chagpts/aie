ITAC Risks at the Input Stage

AI OCR in Accounting Automation

[Study material for accounting professionals. Technical details are simplified for discussion purposes.]

Many companies now use AI-based OCR to read invoices and automatically create journal entries in their accounting systems. While this improves efficiency, it also introduces new IT Application Control (ITAC) risks, especially at the input stage.

Why the Input Stage Matters

The input stage includes:

  • Invoice upload
  • OCR data extraction
  • Document type classification

If data is incorrect at this stage, downstream reviews may not detect the error. This is why auditors often say:

“Garbage in, garbage out.”

Why Auditors Treat This as ITAC

These risks are considered ITAC because:

  • Controls are embedded in the system
  • Errors occur before human review
  • Financial accuracy depends on system logic

Practical Dialogue: Finance Lead vs. OCR Developer

Finance Lead Audit Concern
I’d like to focus on a different risk today. My main concern is document integrity. Once an invoice is uploaded into the system, how can we be sure that it is not replaced or modified later without anyone noticing?

OCR Developer Current Risk
That’s an important point. If there are no controls after upload, there is a risk that a document could be changed later while still looking valid in the system. From a system perspective, it can be difficult to prove that the file was altered.

Finance Lead Audit Reality
Exactly. Auditors usually ask very straightforward questions, such as:

  • Can the uploaded document be replaced?
  • If it is changed, can the system detect it?
  • How do we know the reviewed invoice is the same one that was posted?
If we cannot clearly answer these, they often see it as a control failure.

OCR Developer Basic Control Idea
One common way to address this is to create a digital fingerprint of the document at upload. This fingerprint is called a hash. If the document changes later, the fingerprint also changes, which makes the modification detectable.

OCR Developer Simple Example
Let me give a simple example using MD5. When an invoice is uploaded, the system calculates a hash like this:

Original file MD5: d41d8cd98f00b204e9800998ecf8427e

If even a small part of the file is changed, the MD5 value will be completely different. A mismatch clearly shows that the document is no longer identical to the original upload.

Finance Lead Follow-up Question
That example is easy to understand. But auditors sometimes ask, “Why are you using MD5?” They often recommend SHA-256 instead. Why does that matter?

OCR Developer Audit-Friendly Explanation
MD5 is good for detecting basic or accidental changes, but it is considered weak by today’s standards. In rare cases, two different files can end up with the same MD5 value. Because of that risk, auditors usually prefer SHA-256.

OCR Developer Why SHA-256
SHA-256 creates a much stronger digital fingerprint. In practical terms, this means:

  • It is much harder for two different files to share the same hash
  • It provides stronger evidence that the document was not tampered with
  • Auditors are more comfortable relying on it as a control
So while both MD5 and SHA-256 can detect changes, SHA-256 gives higher assurance when document integrity is critical.

Finance Lead Prevention
Detection is important, but auditors also ask about prevention. Can the document be locked so it cannot be changed after upload?

OCR Developer Read-Only Control
Yes. A common approach is to make the document read-only after upload. Users can view the invoice, but they cannot overwrite or replace the original file. If a correction is needed, a new document must be uploaded instead.

Finance Lead Version History
And when a new document is uploaded, auditors usually want to see a clear history. They ask questions like: “Which version was used for posting?” “Who uploaded the new version?” “Why was the change made?”

OCR Developer Versioning & Logs
That’s handled through document versioning and change logs. Each upload becomes a new version. The system records who uploaded it and when, and the original document is never deleted. The accounting entry clearly links to the specific document version that was used.

Finance Lead Audit Conclusion
From an audit perspective, this is exactly what we need. If we can show that:

  • Documents cannot be silently changed
  • Any change creates a new version
  • Hashes like SHA-256 are used to detect tampering
auditors are much more comfortable relying on the system as a controlled process.

✔ In many audits, document integrity controls (hashing, read-only access, version history, and change logs) are considered more critical than OCR accuracy itself.
