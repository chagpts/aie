ITAC Risks at the Input Stage

AI OCR in Accounting Automation

[Study material for accounting professionals. Technical details are simplified for discussion purposes.]

Many companies now use AI-based OCR to read invoices and automatically create journal entries in their accounting systems. While this improves efficiency, it also introduces new IT Application Control (ITAC) risks, especially at the input stage.

What Is AI OCR?

AI OCR is “intelligent document reading.” Traditional OCR mainly converts images into text. AI OCR goes further by using ML/AI to:

  • Understand layouts (tables, headers, line items, stamps, signatures)
  • Classify documents (invoice vs. receipt vs. credit memo)
  • Extract key fields (vendor, invoice date, total, VAT, currency, bank info)
  • Validate and normalize (date formats, currency symbols, tax formats)
Text recognition Layout understanding Field extraction Document classification Confidence scoring

Types of OCR (and Where AI OCR Fits)

Type How It Works Strengths Limitations
Rule-based OCR
(Template OCR)		 Fixed coordinates or templates per vendor/layout High accuracy for known formats, predictable Breaks when layout changes; scaling vendors is hard
Traditional OCR
(Generic text OCR)		 Converts image → text (no deep understanding) Good for clean scans & simple text blocks Weak on tables/line items; needs heavy post-processing
AI OCR
(Document AI / IDP)		 ML models detect structure + extract fields + classify docs Works across many formats, handles tables better Still error-prone; requires monitoring & governance
Hybrid OCR AI OCR + rules (validation, vendor-specific tweaks) Best practical approach in accounting Needs control design; rules can drift over time

Typical Problems & Risks with AI OCR

Problem What It Looks Like Accounting Risk Control Ideas
“Confidently wrong” extraction Total picked from subtotal; VAT read incorrectly Misstatement in expenses/tax 3-way checks (line total = sum), VAT logic rules, field thresholds
Duplicate / missing invoices Same invoice uploaded twice; missing page Overstatement or incomplete liabilities Duplicate detection (vendor+invoice# + amount), page count checks
Document misclassification Credit memo treated as invoice Wrong direction of posting Doc-type validation, negative amount logic, human review triggers
Security / privacy leakage PII/bank info sent to external OCR service Compliance breach Data masking, vendor due diligence, encryption, retention policies
Model updates change behavior Accuracy drops after vendor release Control failure if unmonitored Change management, regression testing, monitoring dashboards
Audit reality: “AI” does not reduce control needs. It often increases the need for evidence: thresholds, sampling, exception logs, and change management.

Why Use AI OCR in Accounting (Benefits)

  • Speed: faster invoice processing and shorter month-end close
  • Cost: reduces repetitive data entry workload
  • Standardization: normalizes vendor formats into consistent ERP fields
  • Better controls (when designed well): automated validations can be stronger than manual keying
  • Visibility: dashboards for exceptions, aging, and processing bottlenecks

✔ The best results usually come from a hybrid approach: AI OCR + validation rules + human review for low-confidence/high-risk items.

Why the Input Stage Matters

The input stage includes:

  • Invoice upload
  • OCR data extraction
  • Document type classification

If data is incorrect at this stage, downstream reviews may not detect the error. This is why auditors often say:

“Garbage in, garbage out.”

Why Auditors Treat This as ITAC

These risks are considered ITAC because:

  • Controls are embedded in the system
  • Errors occur before human review
  • Financial accuracy depends on system logic

Practical Dialogue: Hash Mismatch Between Interim and Year-End Audit

IT Auditor Interim Audit
During our interim audit, we sampled this transaction and captured the supporting invoice’s SHA-256 hash from your document repository. We’ll re-check the same document during year-end to confirm integrity.

Finance Team Lead Year-end Pressure
Understood. Year-end is busy, but the invoice should be in the system.

IT Auditor Year-end Audit
We pulled the same invoice again for year-end testing. The document ID is the same, but the hash value has changed. That means the binary file is not identical to what existed at interim.

Finance Team Lead Deflection
Could that be just a system update or OCR re-processing?

IT Auditor Technical Proof
OCR re-processing doesn’t change the original file hash. A cryptographic hash changes only if the file content changes. Here’s what we observed: 

Interim audit (SHA-256):  3a7bd3e2360a3d29eea436fcfb7e44c735d117c42d1c1835420b6b9942dd4f1b
Year-end audit (SHA-256): 91b65d1b2c3c7f1c2a6a0fbe0b8b6b9c9c0c8f0b9a7a6b5c4d3e2f1a0b9c8d7e

IT Auditor Control Failure
Same document record, different hash = the document was replaced or modified. This indicates a document integrity control gap: the system allows changes without preserving the original or creating a new version with traceable approval.

OCR Developer Root Cause
Currently, users can overwrite the stored file in the repository. We store metadata, but we don’t enforce immutable storage or versioning, and the ERP link points to the document ID, not a specific version+hash.

Finance Team Lead Admission
At year-end, we swapped in a “cleaner” invoice version to make revenue look stronger. We didn’t realize you had a way to prove the file changed.

IT Auditor Audit Conclusion
This will be raised as an ITAC deficiency. Because the change appears intentional and affects financial reporting, it may also be assessed as a fraud risk and escalated. The remediation expectation is:

  • Hash stored at upload (SHA-256) + monitored for changes
  • Read-only / immutable storage for originals
  • Versioning: any “replacement” becomes a new version
  • Audit logs: who/when/why + approval evidence
  • ERP linkage: posting ties to a specific document version + hash

✔ 핵심: 감사인은 “문서가 있다/없다”가 아니라, 중간감사 때 확인한 그 문서가 기말에도 동일한 파일인지를 본다. 동일 ID인데 hash가 바뀌면, 사실상 “증빙이 바뀐 것”으로 판단된다.
