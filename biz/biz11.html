ITAC Risks at the Input Stage

AI OCR in Accounting Automation

[Study material for accounting professionals. Technical details are simplified for discussion purposes.]

Many companies now use AI-based OCR to read invoices and automatically create journal entries in their accounting systems. While this improves efficiency, it also introduces new IT Application Control (ITAC) risks, especially at the input stage.

Why the Input Stage Matters

The input stage includes:

  • Invoice upload
  • OCR data extraction
  • Document type classification

If data is incorrect at this stage, downstream reviews may not detect the error. This is why auditors often say:

“Garbage in, garbage out.”

Key Input-Stage Risks (ITAC Focus)

✔ Auditors expect controls such as OCR confidence thresholds, mandatory human review, and document integrity checks.

Why Auditors Treat This as ITAC

These risks are considered ITAC because:

  • Controls are embedded in the system
  • Errors occur before human review
  • Financial accuracy depends on system logic

Practical Dialogue: Finance Lead vs. OCR Developer

Finance Lead Audit Concern
I want to shift the discussion a bit. My biggest concern is not only OCR accuracy, but the integrity of the original invoice itself. Once a document is uploaded, how can we be sure it was not replaced or altered later?

OCR Developer Current Setup
At the moment, users upload invoice files into the system, and the OCR process runs on that file. However, I agree that if we do not control what happens after upload, there is a risk.

Finance Lead Audit Reality
Exactly. Auditors often ask very direct questions, like:

  • Can the uploaded document be replaced?
  • If it is changed, can we detect it?
  • How do we know the invoice reviewed is the same one that was posted?
If we cannot answer these clearly, they see it as a serious control weakness.

OCR Developer Risk Explanation
That makes sense. Without proper controls, someone could theoretically upload one invoice, let it pass approval, and then replace the file with a different one later. From the system’s point of view, it would be hard to prove that a change happened.

Finance Lead Key Audit Questions
Right. Auditors usually ask whether we use things like: hash values or checksums, whether the document becomes read-only after upload, and whether there is any version control or change history. If the answer is “no” to all three, they get very uncomfortable.

OCR Developer Simple Controls
To address that, we can introduce some basic but effective controls. First, when a document is uploaded, the system can generate a digital fingerprint (for example, a hash value). If the file is changed later, the fingerprint will no longer match.

Finance Lead Practical Impact
That’s helpful, because then we can show auditors that the invoice used for posting is exactly the same file that was originally uploaded. What about preventing changes in the first place?

OCR Developer Prevention
A common approach is to make the document read-only after upload. Users can still view and reference it, but they cannot overwrite or replace the original file. If a correction is needed, a new document must be uploaded instead.

Finance Lead Version History
And if a new document is uploaded, auditors usually want to see a clear trail. They ask questions like: “Which version was used?” “Why was it changed?” “Who approved the change?”

OCR Developer Change Tracking
That’s where document versioning and change logs come in. Each upload becomes a new version, and the system records who uploaded it and when. The original file is never deleted, and the posting record clearly points to the specific version that was used.

Finance Lead Audit Conclusion
From an audit perspective, that is exactly what we need. If we can demonstrate that:

  • The original document cannot be silently changed
  • Any change creates a new version
  • All changes are logged and reviewable
then auditors are much more comfortable relying on the system.

✔ In audit discussions, document integrity controls are often more important than OCR accuracy itself. Being able to prove that source documents are protected, traceable, and unchanged is a key IT application control.
