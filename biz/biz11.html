ITGC Risks Around OCR & Document Repositories

AI OCR in Accounting Automation

[Study material for accounting professionals. Technical details are simplified for discussion purposes.]

Many companies now use AI-based OCR to read invoices and automatically create journal entries in their accounting systems. While this improves efficiency, it also introduces new IT General Control (ITGC) risks, especially when evidence is stored in a document repository connected to ERP.

What Is AI OCR?

AI OCR is “intelligent document reading.” Traditional OCR mainly converts images into text. AI OCR goes further by using ML/AI to:

  • Understand layouts (tables, headers, line items, stamps, signatures)
  • Classify documents (invoice vs. receipt vs. credit memo)
  • Extract key fields (vendor, invoice date, total, VAT, currency, bank info)
  • Validate and normalize (date formats, currency symbols, tax formats)
Text recognition Layout understanding Field extraction Document classification Confidence scoring

Types of OCR (and Where AI OCR Fits)

Type How It Works Strengths Limitations
Rule-based OCR
(Template OCR)		 Fixed coordinates or templates per vendor/layout High accuracy for known formats, predictable Breaks when layout changes; scaling vendors is hard
Traditional OCR
(Generic text OCR)		 Converts image → text (no deep understanding) Good for clean scans & simple text blocks Weak on tables/line items; needs heavy post-processing
AI OCR
(Document AI / IDP)		 ML models detect structure + extract fields + classify docs Works across many formats, handles tables better Still error-prone; requires monitoring & governance
Hybrid OCR AI OCR + rules (validation, vendor-specific tweaks) Best practical approach in accounting Needs control design; rules can drift over time
Important: OCR accuracy is only half the story. In audits, the bigger question is often whether the supporting documents are protected from tampering through strong ITGCs (access, change, logging, and retention).

Why the Evidence Layer Matters

Even if OCR extracts data perfectly, the audit problem happens when:

  • Users can overwrite a document after upload
  • The system keeps the same document ID but the underlying file changes
  • Logs are incomplete or editable

Auditors often summarize this as:

“If evidence can be changed, evidence is not evidence.”

Key Risks (ITGC Perspective)

ITGC Domain What Can Go Wrong Financial Reporting Impact
Access Management Finance users have privileges to overwrite/replace uploaded invoices.
No SoD (segregation of duties) between uploader, approver, and admin. 		Evidence can be altered to support improper postings (fraud risk).
Change Management Repository/ERP integration changes are deployed without approvals/testing.
Controls (versioning, hashing) can be disabled unintentionally. 		Reliance on system evidence becomes invalid; audit reliance reduced.
IT Operations & Logging Logs are missing, not reviewed, or can be edited/deleted.
No alerting on document overwrite events. 		Inability to prove “who changed what and when” during audit.
Data Integrity / Retention No immutable storage / no retention lock.
Document versioning is absent or optional. 		Audit evidence is not reliable; potential scope expansion and findings.

✔ Practical audit expectation: read-only storage, versioning, hash at upload, and tamper-evident logs are typically assessed as ITGC-related controls over the evidence layer.

Why Auditors Treat This as ITGC

This is viewed as ITGC because the core issue is not “OCR logic,” but whether the company’s IT environment prevents or detects unauthorized changes to systems and records, including:

  • Who can access the repository and what actions they can perform
  • Whether changes are controlled (approvals, testing, deployment controls)
  • Whether operations are monitored (logging, alerting, review)
  • Whether evidence is immutable and retained properly

Practical Dialogue: Hash Mismatch Between Interim and Year-End Audit (ITGC Issue)

IT Auditor Interim Audit
During our interim audit, we sampled this transaction and captured the supporting invoice’s SHA-256 hash from your document repository. We’ll re-check the same document during year-end to confirm integrity.

Finance Team Lead Year-end Pressure
Understood. Year-end is busy, but the invoice should be in the system.

IT Auditor Year-end Audit
We pulled the same invoice again for year-end testing. The document ID is the same, but the hash value has changed. That means the file is not identical to what existed at interim.

Finance Team Lead Deflection
Could that be just a system update or OCR re-processing?

IT Auditor Technical Proof
OCR re-processing doesn’t change the original file hash. A cryptographic hash changes only if the file content changes. Here’s what we observed: 

Interim audit (SHA-256):  3a7bd3e2360a3d29eea436fcfb7e44c735d117c42d1c1835420b6b9942dd4f1b
Year-end audit (SHA-256): 91b65d1b2c3c7f1c2a6a0fbe0b8b6b9c9c0c8f0b9a7a6b5c4d3e2f1a0b9c8d7e

IT Auditor ITGC Finding
Same document record, different hash indicates the document was replaced or modified. This points to an ITGC weakness over the evidence layer: access allows overwrite, and integrity controls (immutable storage/versioning/log review) are not effective.

OCR Developer Root Cause
Users can overwrite the stored file in the repository. We don’t enforce immutable storage or mandatory versioning, and the ERP link points to the document ID, not a specific version+hash.

Finance Team Lead Admission
At year-end, we swapped in a “cleaner” invoice version to make revenue look stronger. We didn’t realize you could prove the file changed.

IT Auditor Audit Conclusion
This will be raised as an ITGC deficiency. Because the change appears intentional and affects financial reporting, it may also be assessed as a fraud risk and escalated. Remediation expectation typically includes:

  • Access control: prevent overwrite; enforce SoD
  • Immutable storage for originals (WORM / retention lock)
  • Versioning: any replacement becomes a new version; originals preserved
  • Hash at upload (SHA-256) stored and monitored for mismatch
  • Tamper-evident logs: who/when/what changed + periodic review
  • ERP linkage: posting ties to a specific document version + hash

Key point: Auditors do not simply check whether a document exists or not. They verify whether the exact same document reviewed at the interim audit is still identical at year-end. If the document ID remains the same but the hash value has changed, auditors conclude that the supporting evidence has been altered.
