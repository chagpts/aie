ITAC Risks at the Input Stage

AI OCR in Accounting Automation

[Study material for accounting professionals. Technical details are simplified for discussion purposes.]

Many companies now use AI-based OCR to read invoices and automatically create journal entries in their accounting systems. While this improves efficiency, it also introduces new IT Application Control (ITAC) risks, especially at the input stage.

What Is AI OCR?

AI OCR is “intelligent document reading.” Traditional OCR mainly converts images into text. AI OCR goes further by using ML/AI to:

  • Understand layouts (tables, headers, line items, stamps, signatures)
  • Classify documents (invoice vs. receipt vs. credit memo)
  • Extract key fields (vendor, invoice date, total, VAT, currency, bank info)
  • Validate and normalize (date formats, currency symbols, tax formats)
Text recognition Layout understanding Field extraction Document classification Confidence scoring

Traditional OCR

(Document AI / IDP)		 ML models detect structure + extract fields + classify docs Works across many formats, handles tables better Still error-prone; requires monitoring & governance

Typical Problems & Risks with AI OCR

Problem What It Looks Like Accounting Risk Control Ideas
“Confidently wrong” extraction Total picked from subtotal; VAT read incorrectly Misstatement in expenses/tax 3-way checks (line total = sum), VAT logic rules, field thresholds
Duplicate / missing invoices Same invoice uploaded twice; missing page Overstatement or incomplete liabilities Duplicate detection (vendor+invoice# + amount), page count checks
Document misclassification Credit memo treated as invoice Wrong direction of posting Doc-type validation, negative amount logic, human review triggers

Audit reality: “AI” does not reduce control needs. It often increases the need for evidence: thresholds, sampling, exception logs, and change management.

Why Use AI OCR in Accounting (Benefits)

  • Speed: faster invoice processing and shorter month-end close
  • Cost: reduces repetitive data entry workload
  • Standardization: normalizes vendor formats into consistent ERP fields
  • Better controls (when designed well): automated validations can be stronger than manual keying
  • Visibility: dashboards for exceptions, aging, and processing bottlenecks

✔ The best results usually come from a hybrid approach: AI OCR + validation rules + human review for low-confidence/high-risk items.

Why the Input Stage Matters

The input stage includes:

  • Invoice upload
  • OCR data extraction
  • Document type classification

If data is incorrect at this stage, downstream reviews may not detect the error. This is why auditors often say:

“Garbage in, garbage out.”

Why Auditors Treat This as ITAC

These risks are considered ITAC because:

  • Controls are embedded in the system
  • Errors occur before human review
  • Financial accuracy depends on system logic

Practical Dialogue: Fake Invoice Swap Detected in ITAC Audit

IT Auditor Walkthrough
For our ITAC walkthrough, we’re tracing one transaction from invoice upload to ERP posting. Please show the original document that supports this journal entry.

Finance Team Lead Year-end Pressure
Sure… just a moment. We needed to hit the year-end numbers, so we updated the invoice file to match the final amount. It’s the same vendor anyway.

IT Auditor Red Flag
When you say “updated,” do you mean the uploaded invoice was replaced after posting? Our concern is whether the system allows a user to swap documents without detection.

Finance Team Lead Admission
The system lets us overwrite the file. At year-end we replaced a few invoices with revised versions so revenue looked higher. We didn’t think it would matter because OCR still reads it.

IT Auditor ITAC Failure
That’s exactly the issue. If an uploaded source document can be replaced, the system has a document integrity gap. In ITAC terms:

  • the “evidence” can be changed after the fact
  • the system cannot prove what was originally uploaded
  • the posting may no longer be supported by the original document
Even if OCR works perfectly, this is a control failure because the system allows tampering.

IT Auditor How It Was Detected
We compared the document timestamp and file metadata, and we also noticed the invoice number format changed. In a stronger system, we would verify the file using a stored SHA-256 hash recorded at upload time. If the hash doesn’t match, the document was altered.

OCR Developer Root Cause
Today we don’t store hashes or enforce versioning. The file can be overwritten in the same document ID, so the ERP entry still “links” to the record even if the binary file changed.

IT Auditor Required Controls
To remediate, we typically expect:

  • Read-only storage after upload (no overwrite)
  • Document versioning (new upload = new version, old preserved)
  • Immutable audit logs (who/when/what changed)
  • Cryptographic hash (e.g., SHA-256) stored at upload
  • Linking of ERP postings to a specific document version + hash
Without these, the system cannot provide reliable audit evidence.

Finance Team Lead Impact
So even though the invoice “exists” in the system, because it can be swapped, auditors treat it as unreliable evidence?

IT Auditor Conclusion
Correct. If documents can be replaced to influence year-end reporting, it becomes both an ITAC deficiency and a potential financial reporting fraud risk. The control objective is simple: the invoice used for posting must be provably the same file that was originally uploaded.

✔ Key audit message: OCR accuracy is not enough. If the system allows silent invoice replacement, the “evidence” is not trustworthy, and auditors will likely flag it as a control deficiency (and escalate if it appears intentional).
